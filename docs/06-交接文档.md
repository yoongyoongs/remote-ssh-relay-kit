当前最大的阻塞不是 Windows 脚本本身，而是公网入口没有完全放通。

证据：

1. 服务器内执行 `curl http://127.0.0.1:8787/health` 成功
2. 服务器内执行 `curl http://106.13.171.166:8787/health` 超时
3. Windows 本机直连 `106.13.171.166:8787` 也超时
4. 当前开发机使用 `ssh2` 连接 `106.13.171.166:22` 时，SSH 握手也超时
5. 现在 `22/TCP` 握手已经恢复正常，但 `8787/TCP` 和 `24000-24999/TCP` 仍然未完成公网验收

所以后续接手时，优先排查：

1. 云安全组
2. 云防火墙
3. 端口白名单
4. 是否真的已放开 `8787/TCP`
5. 是否真的已放开 `24000-24999/TCP`
6. 当前开发机到服务器 `22/TCP` 的链路是否稳定
7. Windows 正式模式是否已经写入 `C:\ProgramData\ssh\administrators_authorized_keys`

1. 修改 `mac/config.ini`
2. 写入正确的 `ENROLL_API`
3. 写入正确的 `ENROLL_CODE`
4. 写入管理员公钥
5. 双击 `start-remote-ssh.command`
6. 观察输出是否成功拿到 `ssh -p 远程端口 用户@106.13.171.166`
7. 在管理员电脑上用私钥发起 SSH

## 7. Windows 后续怎么继续

等公网 `8787` 放通后，Windows 真机建议这样继续：

1. 用本机先测：
   - `curl --noproxy * http://106.13.171.166:8787/health`
2. 成功后再执行：
   - `windows/start-remote-ssh.bat`
3. 检查输出目录中的：
   - `status.json`
   - `result.json`
   - `worker.log`
   - `install-openssh.log`（如果触发安装）
4. 拿到连接命令后，从管理员机器执行 SSH 登录验收
5. 如果目标 Windows 账号属于管理员组，确认 `C:\ProgramData\ssh\administrators_authorized_keys` 已写入管理员公钥
