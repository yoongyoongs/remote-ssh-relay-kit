### 3.14 2026-06-20 修复 Windows 11 管理员公钥文件所有者限制导致的免密登录失败 (成功) [v2-31]

     `& icacls.exe $adminAuthPath /setowner "*S-1-5-32-544" | Out-Null`
2. **测试与重新打包**：

### 3.15 2026-06-20 彻底解决 Windows 免密登录依然需要密码的系统级策略故障 (成功) [v2-32]

为了解决部分 Win11 机器及特定账户在更新 `v2-31` 后依然被要求输入密码的问题，本阶段完成了对系统级安全策略及公钥文件编码污染的彻底清洗与修复，并发布了 `v2-32`：

1. **清洗 UTF-8 BOM 编码污染**：
   - Windows PowerShell 默认使用带 BOM 的 UTF-8（BOM 包含特殊字节标志 `\ufeff`）。这会导致 sshd 将第一行公钥前缀解析为非法字符。本阶段引入了自动读取并以 ASCII 编码写回的清洗逻辑，彻底剥离了已有公钥文件头部的 BOM 字节。
   - 所有后续写入均强制使用 `-Encoding ascii`，保证公钥被 Windows OpenSSH 服务 100% 正确解析。
2. **多用户环境下的严格安全 ACL 控制**：
   - **普通非管理员用户**（或 Match Group 禁用时）：读取 `$env:USERPROFILE\.ssh\authorized_keys`。根据 sshd 的安全限制，除了本用户和 SYSTEM 外，禁止包括 Administrators 组在内的其他任何用户/组拥有写权限。本阶段对该文件增加了严格赋权，剥离了 Administrators。
   - **管理员组用户**：读取 `C:\ProgramData\ssh\administrators_authorized_keys`。本阶段在对文件进行继承权剥离和 Administrators/SYSTEM 赋权外，显式移除了当前运行用户（如 `59404`）自身的显式（非继承）ACE，防止其以显式权限遗留在 ACL 中。
3. **基于 SID 的跨语言系统兼容**：
   - 彻底摒弃以 `$env:USERNAME` 进行授权或移除的常规做法，改用 `[System.Security.Principal.WindowsIdentity]::GetCurrent().User.Value` 获取当前运行用户的唯一 SID，在 icacls 指令中使用语言无关的 `*<SID>` 标识符执行权限管理，完美兼容了中文操作系统及带有空格的用户名。

### 3.17 2026-06-21 修复受控端文件独占锁定与严格权限死锁导致的写入公钥失败 (成功) [v2-34]

在发布 `v2-33` 日志上报版后，部分机器在执行“写入公钥”步骤时报错失败。经深度排查，故障根源在于 Windows 系统下 sshd 服务对公钥文件的独占性锁定（Exclusive Lock）以及上一次运行所应用收紧的 ACL 对写进程产生了权限死锁。

本阶段完成了对该死锁冲突的修复，并发布了 `v2-34`：

1. **共享读写流追加与重试机制 (Safe-Append-Key)**：
   - 编写了 `Safe-Append-Key` 辅助追加函数，采用 .NET `[System.IO.FileShare]::ReadWrite` 共享读写流，在遇到 sshd 进程独占占用的极端情况下依然有读写权限。
   - 增加了重试机制，每次遇到文件占用时自动等待 500ms，重试 4 次，彻底防范了写入崩溃。
2. **写入前重置 ACL 所有权与写入后收紧的闭环机制**：
   - 对普通用户和管理员组两处公钥授权文件，在读取并写入前，**先强行获取文件的所有权 (Owner) 并授予当前提权用户完全控制权 (F)**，以防被上一次运行收紧后的 ACL 锁定导致“拒绝访问”；
   - 在安全追加写入公钥完毕后，**再重新收紧权限并更改 Owner** 以满足 OpenSSH (sshd) 对安全验证的安全检查规范，完美解决了权限死锁。

### 3.18 2026-06-21 引入强力 takeown 夺权、只读清除与防御性公钥写入机制 (成功) [v2-35]

针对部分机器在 `v2-34` 中仍然遇到 `System.UnauthorizedAccessException: 对路径...的访问被拒绝` 的故障，本阶段升级为 `v2-35`：
1. **强行夺取所有权（takeown）打破死锁**：在执行 `icacls` 前，先运行 Windows 系统内置的 `takeown.exe` 强制夺取目录和文件的所有权，从而彻底解开因无所有权导致 `icacls` 静默失败的死锁。
2. **只读属性自动清除**：在文件读写前自动探测并清除可能存在的 ReadOnly（只读）等特殊属性。
3. **防御性公钥写入逻辑**：将写入流程分为“普通用户公钥”和“管理员组公钥”两部分。如果普通用户目录写入失败（可能由防病毒软件或用户目录限制造成），但管理员授权文件写入成功，则记录 Warning 日志并继续执行而不致命中断，极大地增强了对多变系统环境的容错率。
4. **内置 ACL 权限诊断导出**：在写入完成或失败后，自动运行并抓取当前文件的 `icacls` 详细权限列表输出至 `worker.log` 中，使管理员能一步看清权限被拦截的真正根源。

### 3.17 2026-06-21 修复受控端文件独占锁定与严格权限死锁导致的写入公钥失败 (成功) [v2-34]

在发布 `v2-33` 日志上报版后，部分机器在执行“写入公钥”步骤时报错失败。经深度排查，故障根源在于 Windows 系统下 sshd 服务对公钥文件的独占性锁定（Exclusive Lock）以及上一次运行所应用收紧的 ACL 对写进程产生了权限死锁。

本阶段完成了对该死锁冲突的修复，并发布了 `v2-34`：

1. **共享读写流追加与重试机制 (Safe-Append-Key)**：
   - 编写了 `Safe-Append-Key` 辅助追加函数，采用 .NET `[System.IO.FileShare]::ReadWrite` 共享读写流，在遇到 sshd 进程独占占用的极端情况下依然有读写权限。
   - 增加了重试机制，每次遇到文件占用时自动等待 500ms，重试 4 次，彻底防范了写入崩溃。
2. **写入前重置 ACL 所有权与写入后收紧的闭环机制**：
   - 对普通用户和管理员组两处公钥授权文件，在读取并写入前，**先强行获取文件的所有权 (Owner) 并授予当前提权用户完全控制权 (F)**，以防被上一次运行收紧后的 ACL 锁定导致“拒绝访问”；
   - 在安全追加写入公钥完毕后，**再重新收紧权限并更改 Owner** 以满足 OpenSSH (sshd) 对安全验证的安全检查规范，完美解决了权限死锁。

### 3.18 2026-06-21 引入强力 takeown 夺权、只读清除与防御性公钥写入机制 (成功) [v2-35]

针对部分机器在 `v2-34` 中仍然遇到 `System.UnauthorizedAccessException: 对路径...的访问被拒绝` 的故障，本阶段升级为 `v2-35`：
