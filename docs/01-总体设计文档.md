# 总体设计文档

## 1. 目标

本项目的目标是实现这样一套能力：

1. 目标电脑没有公网 IP 也能被远程 SSH 访问。
2. 目标电脑只需要双击一个启动文件即可完成准备工作。
3. 支持 Windows 和 macOS 两类目标系统。
4. 中转服务器不需要把 root 密码下发到客户端。
5. 尽量把风险收敛到最小可控范围。

## 2. 总体架构

整套系统由三部分组成：

1. 操作员端
2. 中转服务器
3. 目标机器

### 2.1 操作员端

操作员端负责：

- 生成自己的 SSH 密钥对
- 部署中转服务器
- 为本次连接准备注册码
- 将配置好的启动包发给目标用户
- 使用最终的 SSH 命令连接目标机器

### 2.2 中转服务器

中转服务器负责：

- 对外提供注册接口
- 为每台目标机器分配唯一远程端口
- 记录设备状态
- 把目标机器的设备公钥写入 `tunnel` 账号的 `authorized_keys`
- 只允许目标机器监听自己被分配到的远程端口

### 2.3 目标机器

目标机器负责：

- 启动本机 SSH 服务
- 写入操作员公钥
- 生成设备密钥
- 调用中转服务器注册接口
- 发起反向 SSH 隧道
- 把最终连接命令展示给目标用户

## 3. 连接流程

### 3.1 注册流程

1. 目标机器读取本地 `config.ini`
2. 目标机器检查本机 SSH 是否可用
3. 目标机器生成设备密钥
4. 目标机器向中转服务器发送注册请求
5. 中转服务器校验注册码
6. 中转服务器分配远程端口
7. 中转服务器返回连接信息

### 3.2 隧道建立流程

1. 目标机器收到远程端口
2. 目标机器执行 `ssh -R`
3. 反向隧道建立到中转服务器
4. 目标机器显示最终 SSH 命令
5. 操作员使用私钥连入目标机器

## 4. 安全设计

## 4.1 为什么不把服务器密码放到客户端

如果把中转服务器的 root 账号密码直接写到 `bat` 或 `sh` 文件里，风险非常高：

- 任何拿到文件的人都能尝试登录服务器
- 用户一旦转发文件，服务器凭据就泄露
- 即使做所谓“加密”，客户端本地也必须能解开，本质上仍可被提取

因此本方案明确不把服务器登录密码下发给客户端。

## 4.2 客户端实际持有的内容

客户端仅持有：

- 中转服务器地址
- 注册接口地址
- 一次性的注册码
- 操作员 SSH 公钥

这些信息即使泄露，也只会影响某个注册码周期，不会直接暴露服务器 root 权限。

## 4.3 中转服务器的 SSH 限制

目标机器连接中转服务器使用的是专门的 `tunnel` 账号。

限制思路如下：

- 不给该账号 sudo 权限
- 不让它承载业务
- 在 `authorized_keys` 中限制可监听端口
- 在 `sshd_config` 中只允许远程转发
- 禁止 TTY、Agent Forwarding、X11 Forwarding 等无关能力

## 4.4 操作员如何连接目标机器

操作员不是去登录中转服务器，而是通过中转服务器暴露出来的远程端口，直接连到目标机器的本地 SSH 服务。

例如：

```bash
ssh -i admin_ed25519 -p 24001 Alice@106.13.171.166
```

这里：

- `24001` 是中转服务器分配给目标机器的远程端口
- `Alice` 是目标机器上的本地用户名
- `106.13.171.166` 是中转服务器公网地址

## 5. 目录与组件设计

## 5.1 server

- `relay-server.mjs`：注册 API
- `install-server.sh`：Linux 安装脚本
- `relay.env.sample`：环境变量模板
- `sshd_config.sample`：SSH 限制配置示例

## 5.2 windows

- `start-remote-ssh.bat`：一键启动入口
- `RemoteSshApp.ps1`：可视化控制器
- `RemoteSshWorker.ps1`：提权后的执行器
- `config.ini`：客户端配置

## 5.3 mac

- `start-remote-ssh.command`：一键启动入口
- `remote-ssh.sh`：执行脚本
- `config.ini`：客户端配置

## 5.4 scripts

- `prepare-launchers.ps1`：把注册码、公钥和服务器地址写入启动包
- `build-release.ps1`：构建压缩包
- `deploy-relay.mjs`：把中转服务部署到 Linux 服务器

## 6. 状态管理设计

中转服务器维护一个 `devices.json`，记录：

- 设备 ID
- 设备名
- 系统类型
- 本地用户名
- 设备公钥
- 分配到的远程端口
- 当前状态
- 创建时间
- 最后心跳时间
- 过期时间

同时会根据当前有效设备，重建 `authorized_keys`。

## 7. 已知边界与限制

1. 当前 Windows 启动器主要围绕 OpenSSH Server 方案实现。
2. macOS 依赖系统自带 Remote Login/ssh 服务。
3. 现在已经实现注册、隧道启动、结果展示，但没有做完整守护式断线重连。
4. 注册码如果长期不更换，安全性会下降，因此建议按批次轮换。

## 8. 适用场景

适合：

- 远程协助开发机
- 临时接入测试电脑
- 家用网络环境下的远程排障
- 无公网 IP 场景下的 SSH 接入

不适合：

- 高强度生产跳板机场景
- 需要完整审计、权限分级、多租户管理的企业平台
- 对长期稳定隧道和自动重连要求非常高的商业场景
